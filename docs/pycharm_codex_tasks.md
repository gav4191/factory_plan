# Завдання для Codex у PyCharm

Цей файл потрібен, щоб продовжити розробку в PyCharm без втрати контексту.

## Важливо

Не змінювати бізнес-правила без погодження з користувачем. Поточні правила зафіксовані у файлах:

- `docs/decisions.md`
- `docs/data_model.md`
- `docs/scheduling.md`
- `docs/interface.md`

## Поточний стан

Створено базовий FastAPI-каркас:

- `main.py`
- `requirements.txt`
- `README.md`
- `factory_plan/app.py`
- `factory_plan/database.py`
- `factory_plan/importer.py`
- `factory_plan/routers/api.py`
- `factory_plan/templates/*.html`
- `factory_plan/static/app.css`
- `factory_plan/static/app.js`

Залежності ще не встановлені. Їх має встановити Codex у PyCharm або розробник вручну.

## Перший запуск

1. Встановити залежності:

```powershell
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

2. Запустити сервер:

```powershell
.\.venv\Scripts\python.exe -m uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

3. Відкрити:

```text
http://127.0.0.1:8000
```

Для локальної мережі:

```text
http://<IP-сервера>:8000
```

## Найближчі задачі

### 1. Доробити імпорт

Поточний endpoint:

```text
POST /api/import/preview
```

Він тільки читає файл і показує preview.

Потрібно додати:

```text
POST /api/import
```

Логіка:

1. Прийняти TSV/CSV файл.
2. Розпізнати колонки.
3. Очистити значення від зайвих пробілів.
4. Перетворити дати у формат ISO `YYYY-MM-DD`.
5. Перетворити `Так`/`Ні` у boolean.
6. Перетворити години з комою у float.
7. Записати двері у `doors`.
8. Записати операції у `operations`.
9. Якщо МДФ має `Ні`, але час порожній, створити операцію як `is_required = 0` або не створювати її. Обрати один підхід і зафіксувати в `docs/data_model.md`.
10. Повернути кількість імпортованих дверей і попередження.

### 2. Додати CRUD для календаря

Потрібні API:

```text
GET /api/calendar
PUT /api/calendar/{date}
POST /api/calendar/generate
```

Мінімальна логіка:

- дата
- робочий/вихідний
- кількість годин

Генерація календаря може створювати дні від сьогодні на 90 або 120 днів вперед.

### 3. Додати CRUD для правил черги

Потрібні API:

```text
GET /api/priorities
POST /api/priorities
DELETE /api/priorities/{id}
```

Поля:

- `rule_type`: `serial`, `order_number`, `counterparty`
- `rule_value`
- `priority`
- `comment`

Правило сортування:

1. менший priority іде раніше
2. ручний priority повністю перебиває дати
3. конкретність: serial > order_number > counterparty

### 4. Реалізувати планувальник

Створити модуль:

```text
factory_plan/scheduler.py
```

Публічна функція:

```python
def recalculate_plan(start_date: date | None = None) -> None:
    ...
```

Вона має:

1. Очистити `plan_items` і `plan_daily_load`.
2. Взяти старт від сьогодні, якщо `start_date is None`.
3. Побудувати чергу дверей.
4. Для кожної двері визначити наступні незавершені операції.
5. Планувати тільки робочі дні з календаря.
6. Для кожного етапу на день використовувати ємність `work_calendar.hours`.
7. Дозволяти переповнення тільки останньою операцією дня.
8. Не планувати більше одного етапу однієї двері в один день.
9. Враховувати залежності:
   - Заготовка -> Зварка -> Фарбування
   - МДФ окремо, якщо потрібен
   - Зборка після завершення A і МДФ, якщо МДФ потрібен
   - Упаковка після Зборки
10. Записати результат у `plan_items` і `plan_daily_load`.

### 5. Додати endpoint перерахунку

```text
POST /api/plan/recalculate
```

Після виклику має запускати `recalculate_plan()` і повертати коротку статистику:

- кількість операцій у плані
- кількість днів
- кількість переповнених комірок день/етап

### 6. Оживити сторінки

Після API доробити frontend:

- `/import`: кнопка не тільки preview, а імпорт
- `/calendar`: редагування днів і годин
- `/priorities`: форма додавання і видалення правил
- `/`: кнопка перерахунку плану
- `/doors`: фільтри

## Тестові сценарії

Після реалізації імпорту і планувальника перевірити:

1. Імпорт файлу `для плану виконання.txt`.
2. Рядки з `МДФГотово = Ні`, але порожнім `Час МДФ`, не повинні планувати МДФ.
3. Двері не проходить два етапи в один день.
4. Зборка не стартує до завершення фарбування і МДФ, якщо МДФ потрібен.
5. Якщо операція перевищує залишок дня, вона додається останньою і день закривається.
6. Ручний пріоритет по серії перемагає правило по замовленню і контрагенту.
7. Правило по замовленню перемагає правило по контрагенту.
