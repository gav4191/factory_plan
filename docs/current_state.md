# Поточний стан роботи

Дата фіксації: 2026-05-18

## Що вирішили

Основний формат обміну з 1С - JSON.

1С є джерелом правди для даних по дверях. Якщо при імпорті `serial` вже існує в базі Factory Plan, локальні дані цієї двері оновлюються з JSON:

- оновлюється запис у `doors`
- старі операції цієї двері видаляються з `operations`
- операції створюються заново з JSON

Ручні зміни після виконання робіт мають потрапляти назад у 1С. Наступний імпорт з 1С може перезаписувати локальні дані.

## Канали імпорту

Підтримуються два шляхи:

1. Пряма передача JSON з 1С через API:

```text
POST /api/import/json
```

2. Завантаження JSON з каталогу:

```text
data/import/inbox
```

Через endpoint:

```text
POST /api/import/from-folder
```

Після успішного імпорту файл переноситься в:

```text
data/import/archive
```

Якщо файл з помилкою - в:

```text
data/import/error
```

Автотригер на появу файла поки не робимо. Імпорт з каталогу запускається вручну.

## Що вже реалізовано

У коді є:

- `POST /api/import/json`
- `POST /api/import/file`
- `POST /api/import/from-folder`
- `POST /api/doors/clear`
- кнопка `Очистити двері і план` на сторінці `/import`

Кнопка очищення має чистити дані, що залежать від дверей:

- `plan_daily_load`
- `plan_items`
- `operations`
- `doors`

І не має чистити:

- `work_calendar`
- `priority_rules`

## Перевірений файл імпорту

Файл був у:

```text
data/import/inbox/factory_plan_doors_20260518_212225.json
```

Після імпорту був перенесений у:

```text
data/import/archive/factory_plan_doors_20260518_212225_2026-05-18_213555.json
```

Перевірка JSON показала:

- JSON валідний
- `format = factory_plan_doors`
- `version = 1`
- `generated_at = 2026-05-18T21:22:50+03:00`
- дверей у файлі: `1618`
- унікальних серій: `1618`
- критичних помилок: `0`
- попереджень: `0`
- орієнтовно операцій до планування: `5721`
- МДФ без годин, який треба пропускати як непотрібний: `49`

У файлі є бізнесово підозріла дата `planned_date = 2030-06-25`, але це не помилка імпорту. Бізнес-перевірки дат вирішили робити пізніше на етапі роботи з планом, не під час імпорту.

## Стан після імпорту, який перевіряли напряму в SQLite

Після імпорту база містила:

- `doors`: `1686`
- `operations`: `10066`
- дублікати `serial`: `0`
- двері без операцій: `0`
- `plan_items`: `6275`
- `work_calendar`: `120`
- `priority_rules`: `0`

Чому `doors = 1686`, а не `1618`: раніше вже був імпортований тестовий `sample_ok_2026-05-18_204831.json`.

## Важливий момент

Під час ручної спроби очистити базу напряму через SQLite виник `sqlite3.OperationalError: disk I/O error`. Перед цим було створено backup:

```text
data/factory_plan_before_clear_2026-05-18_214043.sqlite3
```

Після цього вирішили не чистити базу руками, а користуватися штатним очищенням через `POST /api/doors/clear` або кнопку на `/import`.

## Наступний крок

Перевірити штатний сценарій:

1. Натиснути кнопку `Очистити двері і план` на `/import`.
2. Переконатися, що очистилися `doors`, `operations`, `plan_items`, `plan_daily_load`.
3. Переконатися, що `work_calendar` і `priority_rules` залишилися.
4. Повторно імпортувати великий JSON-файл з архіву або новий файл з 1С.
5. Після чистого імпорту перейти до перевірки/доробки планувальника.
