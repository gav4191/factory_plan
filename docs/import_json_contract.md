# Контракт JSON-імпорту з 1С

## Призначення

1С формує JSON-файл зі списком дверей для виробничого планування. Factory Plan імпортує цей файл, створює або оновлює картки дверей та операції виробничого маршруту.

Файл має бути у кодуванні UTF-8 без BOM або UTF-8 з BOM. Рекомендоване розширення: `.json`.

## Загальна структура файла

```json
{
  "format": "factory_plan_doors",
  "version": 1,
  "generated_at": "2026-05-18T15:30:00+03:00",
  "source": {
    "system": "1C",
    "database": "production",
    "organization": "Назва організації"
  },
  "doors": []
}
```

## Шаблон однієї двері

```json
{
  "serial": "A-000123",
  "order_date": "2026-05-10",
  "planned_date": "2026-05-25",
  "counterparty": "ТОВ Приклад",
  "order_number": "ЗМ-4567",
  "door_type": "Вхідні",
  "model": "Модель 21",
  "size": "960x2050",
  "leaf_count": 1,
  "is_custom": false,
  "operations": {
    "blank": {
      "done": false,
      "hours": 1.5
    },
    "welding": {
      "done": false,
      "hours": 2.0
    },
    "painting": {
      "done": false,
      "hours": 1.25
    },
    "mdf": {
      "done": false,
      "hours": null
    },
    "assembly": {
      "done": false,
      "hours": 1.75
    },
    "packing": {
      "done": false,
      "hours": 0.5
    }
  }
}
```

## Повний приклад файла

Дивись файл [import_sample.json](import_sample.json).

## Обов'язкові поля

На рівні кореня:

- `format`: завжди `factory_plan_doors`
- `version`: ціле число, поточна версія `1`
- `generated_at`: дата і час формування файла у форматі ISO 8601
- `doors`: масив дверей

На рівні двері:

- `serial`: унікальна серія двері
- `order_date`: дата замовлення
- `planned_date`: планова дата виготовлення
- `counterparty`: контрагент
- `order_number`: номер замовлення
- `door_type`: тип двері
- `model`: модель
- `size`: розмір
- `leaf_count`: кількість створок
- `operations`: об'єкт з етапами

## Необов'язкові поля

- `source`: службова інформація про джерело
- `is_custom`: ознака нестандартної двері, якщо поля немає - вважати `false`

## Операції

У `operations` використовуються тільки такі ключі:

- `blank`: Заготовка, лінія A, порядок 1
- `welding`: Зварка, лінія A, порядок 2
- `painting`: Фарбування, лінія A, порядок 3
- `mdf`: МДФ, лінія B, порядок 1
- `assembly`: Зборка, лінія C, порядок 1
- `packing`: Упаковка, лінія C, порядок 2

Для кожної операції:

- `done`: boolean, `true` якщо етап вже виконаний, `false` якщо ще ні
- `hours`: число годин або `null`

Якщо етап виконаний (`done = true`), він не планується. `hours` для виконаного етапу може бути числом або `null`.

Якщо етап не виконаний (`done = false`) і `hours` більше `0`, етап створюється як потрібний для планування.

Якщо `mdf.done = false`, але `mdf.hours = null` або `0`, МДФ для цієї двері вважається непотрібним і не планується.

Для інших невиконаних етапів бажано передавати `hours > 0`. Якщо `hours` порожній або `0`, імпорт має видати попередження.

## Формати значень

Дати:

- формат тільки `YYYY-MM-DD`
- приклад: `2026-05-18`
- не передавати дати у форматі `18.05.2026`

Дата і час формування:

- формат ISO 8601
- приклад: `2026-05-18T15:30:00+03:00`

Числа:

- години передавати числом JSON, не рядком
- дробова частина через крапку: `1.5`, не `1,5`
- не передавати одиниці виміру в полі: `1.5`, не `"1.5 год"`

Boolean:

- тільки JSON-значення `true` або `false`
- не передавати `"Так"`, `"Ні"`, `"Да"`, `"Нет"`, `1`, `0`

Текст:

- обрізати зайві пробіли на початку і в кінці
- не передавати службові символи табуляції та переноси рядків всередині простих текстових полів
- порожні необов'язкові текстові поля передавати як `null` або порожній рядок

## Правила ідентифікації при повторному імпорті

Головний ключ двері - `serial`.

Якщо двері з таким `serial` вже є в базі Factory Plan, імпорт оновлює її реквізити та операції. Якщо двері немає - створює нову.

У межах одного файла `serial` не має повторюватися. Повтор серії в одному файлі є помилкою файла.

## Вимоги до розробника 1С

1. Формувати валідний JSON, який відкривається стандартним JSON-парсером без виправлень вручну.
2. Використовувати UTF-8.
3. Передавати всі обов'язкові поля для кожної двері.
4. Передавати дати тільки у форматі `YYYY-MM-DD`.
5. Передавати години числом, з крапкою як десятковим розділювачем.
6. Передавати ознаки виконання етапів як `true` або `false`.
7. Не локалізувати ключі JSON. Ключі мають бути саме англійські: `serial`, `order_date`, `operations`, `blank`, `welding` тощо.
8. Не змінювати назви ключів без погодження, бо це ламає імпорт.
9. Не передавати зайві вкладені структури всередині двері, якщо вони не описані в цьому контракті.
10. Забезпечити унікальність `serial` у межах файла.
11. Якщо для двері МДФ не потрібен, передавати `mdf.hours` як `null` або `0`.
12. Якщо етап вже виконаний у 1С, передавати `done: true`, щоб Factory Plan не планував його повторно.
13. Якщо значення невідоме, краще передати `null`, а не текст `"не вказано"` чи `"-"`.

## Мінімальні перевірки перед вивантаженням з 1С

Перед записом файла 1С має перевірити:

- `doors` не порожній
- всі `serial` заповнені
- немає повторів `serial`
- всі дати заповнені та відповідають формату `YYYY-MM-DD`
- `leaf_count` є цілим числом більше `0`
- всі невиконані обов'язкові операції мають додатні години
- всі ключі операцій входять у дозволений список

## Очікувана поведінка імпорту у Factory Plan

Імпорт має повернути:

- кількість створених дверей
- кількість оновлених дверей
- кількість створених або оновлених операцій
- список помилок, через які файл не можна імпортувати
- список попереджень, які не блокують імпорт

Якщо є критичні помилки структури або обов'язкових полів, файл не імпортується частково. Користувач отримує список помилок і виправляє файл у 1С.
