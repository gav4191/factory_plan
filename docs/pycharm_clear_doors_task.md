# Завдання для PyCharm: кнопка очищення даних по дверях

## Мета

Додати в Factory Plan безпечну можливість очистити всі дані, які стосуються імпортованих дверей і побудованого плану, щоб можна було заново імпортувати файл з 1С.

Очищення має запускатися вручну з інтерфейсу кнопкою.

## Що потрібно очищати

Очистити таблиці:

- `plan_daily_load`
- `plan_items`
- `operations`
- `doors`

Це дані, які напряму залежать від імпортованих дверей.

## Що НЕ очищати

Не очищати:

- `work_calendar`
- `priority_rules`

Причина:

- календар є налаштуванням виробничої потужності
- правила пріоритету є ручними налаштуваннями черги

Якщо пізніше буде потрібно очищати і їх, це має бути окрема дія з окремим підтвердженням.

## Backend

Додати endpoint:

```text
POST /api/doors/clear
```

Endpoint має:

1. Приймати запит без тіла.
2. Виконувати очищення в одній транзакції.
3. Видаляти таблиці у правильному порядку:

```text
plan_daily_load
plan_items
operations
doors
```

4. Не чіпати `work_calendar` і `priority_rules`.
5. Повернути кількість видалених записів по кожній таблиці.

Приклад відповіді:

```json
{
  "success": true,
  "deleted": {
    "plan_daily_load": 720,
    "plan_items": 6275,
    "operations": 10066,
    "doors": 1686
  }
}
```

Якщо сталася помилка, транзакція має відкотитися.

## Важливо щодо SQLite

Не обов'язково скидати `sqlite_sequence`. Для цієї задачі достатньо видалити записи.

Не використовувати видалення файла бази.

Не очищати всю базу.

## Frontend

На сторінці `/import` додати окремий блок або кнопку:

```text
Очистити двері і план
```

Поведінка:

1. При натисканні показати підтвердження через `confirm()`.
2. Текст підтвердження:

```text
Видалити всі імпортовані двері, операції та розрахований план? Календар і правила черги залишаться.
```

3. Якщо користувач підтвердив - викликати:

```text
POST /api/doors/clear
```

4. Після успіху показати результат:

```text
Видалено: дверей 1686, операцій 10066, записів плану 6275, завантажень днів 720.
```

5. Якщо помилка - показати повідомлення з текстом помилки.

## Де краще розмістити

Розмістити кнопку на сторінці `/import`, бо очищення зазвичай потрібне перед повторним імпортом.

Кнопка має бути візуально небезпечною, наприклад червона/secondary danger, але без складного дизайну.

## Додаткова перевірка перед очищенням

Перед видаленням endpoint може порахувати записи в таблицях і потім видалити їх.

Псевдологіка:

```python
tables = ["plan_daily_load", "plan_items", "operations", "doors"]
deleted = {}
with get_connection() as connection:
    connection.execute("BEGIN")
    for table in tables:
        deleted[table] = connection.execute(f"SELECT COUNT(*) FROM {table}").fetchone()[0]
    for table in tables:
        connection.execute(f"DELETE FROM {table}")
    connection.commit()
return {"success": True, "deleted": deleted}
```

Реалізувати без SQL-ін'єкції: список таблиць має бути жорстко заданий у коді, не приймати назви таблиць від користувача.

## Тести вручну

1. Перед очищенням перевірити, що `/api/doors` повертає двері.
2. Натиснути кнопку `Очистити двері і план`.
3. Підтвердити дію.
4. Перевірити відповідь API.
5. Перевірити, що:
   - `doors` порожня
   - `operations` порожня
   - `plan_items` порожня
   - `plan_daily_load` порожня
   - `work_calendar` залишилась без змін
   - `priority_rules` залишилась без змін
6. Після очищення покласти JSON у `data/import/inbox` або завантажити файл через UI і виконати імпорт заново.

## Критерії готовності

Завдання готове, якщо:

- є `POST /api/doors/clear`
- endpoint очищає тільки `plan_daily_load`, `plan_items`, `operations`, `doors`
- календар і правила черги не видаляються
- очищення виконується транзакційно
- `/import` має кнопку очищення з підтвердженням
- після очищення можна повторно імпортувати JSON з 1С
